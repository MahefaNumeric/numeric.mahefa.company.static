Code source HTML : restez léger !

Pour afficher une page Web, le code source HTML est la première chose que les navigateurs Web téléchargent, analysent et modélisent. Si la quantité de code contenue dans la page est trop importante, ces étapes sont ralenties et le rendu est retardé.

Avec notre prochaine mise à jour, nous commencerons à vérifier les tailles de votre page HTML. “Sizes” est au pluriel, car nous vérifierons à la fois la taille compressée (par exemple: gzip) et la taille non compressée.
Avoir une taille compressée supérieure à 200 Ko sera considéré comme un problème majeur. Idem pour une taille non compressée supérieure à 1 Mo.

Versions jQuery non prises en charge

jQuery 1.X n’est plus pris en charge. Pour les sites utilisant l’une de ces versions, nous recommandons – au moins – de passer à la dernière disponible sur la branche 1.X : jQuery 1.12.
Même s’il n’est pas pris en charge non plus, il corrige de nombreux bugs et problèmes de sécurité. Néanmoins, la solution idéale serait de passer à la dernière version majeure (ou d’utiliser un autre framework à jour).

Sécurisation des liens relatifs avec l’élément de base et la directive CSP base-uri

Nous avons beaucoup écrit sur la politique de sécurité du contenu sur ce blog afin de protéger vos visiteurs des effets d’une attaque XSS.
Des pages contenant des liens relatifs pourraient être détournées lors d’une telle attaque. Nous conseille d’utiliser l’élément pour les pages utilisant des liens relatifs. Ce n’est qu’une première étape, et un attaquant pourrait toujours modifier la valeur de base s’il y a une violation XSS (ou ajouter une autre balise ).
Pour éviter cela, vous pouvez appliquer la valeur de l’attribut href en utilisant la directive CSP base-uri.

Gardez vos pages conviviales pour les applications de synthèse vocale

Avoir une page accessible à tous est une question de bon sens. C’est aussi une obligation légale dans de plus en plus de pays.

La plupart des logiciels de synthèse vocale ont des problèmes lorsqu’ils essaient de vocaliser des éléments HTML vides. Nous ajouterons de nouveaux contrôles de qualité pour détecter les balises HTML problématiques.

Fichiers GIF, à utiliser à bon escient

Vous avez peut-être déjà vu notre recommandation d’utiliser les formats PNG ou JPEG pour les images GIF non animées. Nous allons ajouter un nouveau chèque, cette fois sur GIF animé.
La prise en charge du MP4 est désormais quasi universelle, Nous recommandera donc d’utiliser le format vidéo plutôt que GIF, pour les tailles de fichiers supérieures à 100 Ko.

Des dizaines d’améliorations pour les contrôles de qualité existants

Nous avons amélioré et ajouté quelques détails sur certaines bonnes pratiques (merci à tous les utilisateurs qui nous ont envoyé des commentaires !). Nous ajusterons également la pénalité appliquée pour certains contrôles de qualité, afin de rendre les priorités de notre rapport toujours plus pertinentes.

Le périmètre de plusieurs bonnes pratiques sera également élargi : redirections côté client, lazy-load d’images, utilisation de la meta viewport, utilisation du flash, etc.

Nous détecterons le SPOF pour la liste suivante de fournisseurs de contenu tiers :

Nous avons également corrigé la vérification X-Content-Type-Options pour ignorer les images, car les spécifications de l’en-tête ont évolué. Merci à @nhoizey et @etportis pour l’avoir signalé.

Nous étendrons notre recommandation d’utiliser un système de mise en cache lorsque nous détectons un temps de réponse du serveur très lent (supérieur à 3 secondes). Ce test était auparavant appliqué uniquement pour les CMS WordPress et Prestashop.

Enfin, toutes les vérifications des meilleures pratiques d’accessibilité seront basées sur le DOM plutôt que sur le code source HTML, car la plupart des logiciels et appareils prennent désormais en charge Javascript.
